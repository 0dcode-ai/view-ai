# 产品路线图

更新时间：2026-05-08

## 产品定位

项目当前定位是“面试准备和训练 SaaS”。它不是单纯题库，也不是单纯模拟面试，而是围绕一个具体求职机会，把简历、JD、技术知识、开源项目、面经、模拟练习、复盘和下一步行动串成闭环。

核心主线：

```text
求职机会 -> 来源材料 -> Agent 生成准备内容 -> 模拟练习 -> 复盘行动 -> 下一轮训练
```

## 当前阶段

项目处于“产品主线收束 + 后端迁移并行”的阶段。

已经具备：

- 求职机会工作台
- 简历 / JD 管理
- 简历版本和 JD 匹配
- 八股知识库和题解式学习稿
- 技术文章库
- 面试官 Agent
- 面试者练习和候选人准备 Agent
- 面经和公司情报
- 复盘卡和冲刺计划
- 开源趋势和 GitHub 来源材料
- 独立 Agent 配置和运行日志
- Next Web + Nest API + Postgres 的过渡架构

主要债务：

- Web 端仍保留大量 Next API routes，部分功能已迁到 `apps/api`。
- 同一能力在 Next API 和 Nest API 之间仍有重复实现。
- Agent 协议已经有雏形，但历史 Agent 未完全统一。
- 数据源仍是双轨：Web SQLite + API Postgres。
- UI 功能密度较高，需要进一步围绕“当前求职机会”收敛。

## 阶段 1：求职机会主线打穿

目标：让用户进入产品后，默认围绕一个 Application 完成准备、练习和复盘。

### 需要继续做

- 首页“今日”优先展示高优先级 Application 的下一步动作。
- 求职机会详情里增加“准备进度解释”，告诉用户为什么是这个分数。
- Application 详情里把候选人准备 Agent、面试官 Agent、模拟练习、复盘卡串成明确按钮链路。
- 面试结束后自动回写 Application：
  - 最近模拟分
  - 复盘完成度
  - 下一步动作
  - 风险点
- ReviewCard 和 SprintPlan 默认关联当前 Application。

### 验收标准

- 新建一个求职机会后，不需要跳多个页面也能完成：粘贴 JD、生成匹配、创建简历版本、启动候选人准备、开始模拟、查看复盘。
- 刷新页面后仍保留当前页签和当前 Application。
- 无简历、无 JD、有简历无 JD、有简历有 JD 四种状态都有清晰空状态。

## 阶段 2：Agent 完全独立运行

目标：每个 Agent 都可配置、可运行、可观察、可替换模型。

### 需要继续做

- 统一所有 Agent 的输入 schema、输出 schema、fallback 和 execution。
- 把历史 Next API Agent 逐步迁到 `/v1/agents/:agentName/run`。
- 为每个 Agent 增加 prompt 版本号。
- AgentRunLog 增加关联资源：
  - applicationId
  - resumeProfileId
  - jobTargetId
  - sourceDocumentId
  - knowledgeCardId
- 前端 Agent 设置页支持查看最近运行、失败原因和 fallback 状态。

### 优先迁移顺序

1. `knowledge-record`
2. `candidate-prep`
3. `application-match`
4. `resume-tailor`
5. `mock-interviewer`
6. `github-repo`
7. `startup-idea`

### 验收标准

- 任意 Agent 失败都不会阻断主流程，前端能展示可编辑 fallback。
- 每次 Agent 运行都可在日志里查到输入、输出、模型、耗时、错误和证据。
- 每个 Agent 可单独改模型配置，不影响其他 Agent。

## 阶段 3：知识库产品化

目标：把八股、技术文章、面经和开源项目统一成可复习、可引用、可生成面试材料的知识资产。

### 已完成基础

- 八股卡已升级为题解式学习稿。
- 技术文章可保存、编辑、上传图片。
- 文章拆卡可把长文转为多张八股学习稿。
- 开源趋势可保存为来源材料。

### 需要继续做

- 新增统一“知识入库 Agent”，支持：
  - 粘贴长文
  - 选择技术文章
  - 选择 GitHub 来源
  - 选择面经
  - 生成八股卡、面试卡、项目讲述卡
- 给 KnowledgeCard 增加更明确的来源引用。
- 技术文章支持 OCR 的设计先预留，不急着实现。
- 复习模式按“题解式学习稿”拆步骤训练：
  - 先看题目
  - 自己说一句话结论
  - 展开思路
  - 回答追问
  - 对照复习清单

### 验收标准

- 用户贴一整段文章后，可以稳定生成多张结构化学习稿。
- 每张卡都能看到来源、结论、思路、追问和复习清单。
- 知识卡可以被模拟面试和候选人准备 Agent 引用。

## 阶段 4：后端统一到 Postgres

目标：逐步让 Nest API + Postgres 成为唯一业务后端，Next Web 只保留页面和少量兼容代理。

详细资源状态见：

```text
docs/资源迁移状态.md
```

### 当前状态

- `apps/api` 已使用 Postgres。
- Web 根目录仍有 SQLite Prisma schema。
- `app/api-client.ts` 已支持按资源代理到外部 `/v1` API。
- `scripts/dev-all.js` 可以一键启动 Postgres、Web 和 API，并同步 schema。

### 需要继续做

- 明确每个资源的迁移状态：
  - 已迁移
  - 部分迁移
  - 仍走 Next API
- 技术文章、创业想法、开源趋势等仍需决定是否迁入 `apps/api`。
- 把 Web 端对于已迁移资源的 Next API route 标记为兼容层，避免继续扩写重复业务。
- 补 API service 单测，优先覆盖 Application、Knowledge、Agent、Interview。

### 验收标准

- `NEXT_PUBLIC_API_BASE_URL=http://localhost:4000/v1` 时，Web 核心链路能跑通。
- 断开 API 时，文档明确哪些功能还能走本地 Next API，哪些功能不可用。
- Postgres schema 成为生产环境唯一可信 schema。

## 阶段 5：演示与生产基础

目标：让项目既能稳定演示，也能继续向真实产品推进。

### 需要继续做

- Web 首页整理成“今日 + 当前求职机会”的清晰入口。
- 移动端继续保留纯前端演示，不急着承接复杂真实数据。
- 增加种子数据脚本，能一键生成演示用：
  - 求职机会
  - 简历
  - JD
  - 八股卡
  - 面试记录
  - 开源趋势来源
- 补充错误状态：
  - GLM 余额不足
  - GitHub 限流
  - Postgres 未启动
  - API 未启动
- 增加基础监控字段：
  - Agent latency
  - usedFallback
  - rate limit
  - last successful sync

### 验收标准

- `npm run dev:all` 后，Web 和 API 能稳定启动。
- 演示链路不依赖手动造数据。
- 模型或 GitHub 失败时，页面有清晰提示，不会白屏。

## 当前优先级建议

下一步优先级：

1. 把 Application 详情的闭环补强，让求职机会成为真正主线。
2. 把 `knowledge-record` Agent 接入统一 Agent 协议。
3. 给知识卡增加来源引用，让 Agent 输出能追溯。
4. 梳理资源迁移表，明确哪些走 Postgres，哪些仍走 SQLite。
5. 优化首页，让“今天该做什么”直接来自当前 Application。

## 暂不做

- 不做真实面试中的隐身实时辅助。
- 不急着做 Chrome 扩展。
- 不急着做移动端真实 API 全接入。
- 不急着做复杂权限和团队协作。
- 不急着做 PDF/DOCX 简历导出，等简历版本流稳定后再做。
