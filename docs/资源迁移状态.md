# 资源迁移状态

更新时间：2026-05-08

## 目的

这份文档记录 Web Next API、Nest API 和数据库之间的迁移状态。当前项目处于过渡阶段：Web 端仍有 Next API routes 和 SQLite，`apps/api` 已使用 NestJS + Postgres，Web 通过 `app/api-client.ts` 按资源代理到 `/v1/*`。

## 当前代理配置

Web 端通过下面两个环境变量控制是否走独立 API：

```text
NEXT_PUBLIC_API_BASE_URL="http://localhost:4000/v1"
NEXT_PUBLIC_API_MIGRATED_RESOURCES="knowledge,resumes,job-targets,interviews,interviewer-sessions,reviews,sprints,daily,experiences,companies,labs,learning-paths,applications,resume-versions,sources,agents"
```

如果 `NEXT_PUBLIC_API_BASE_URL` 为空，则 Web 端全部走本地 Next API routes。

如果 `NEXT_PUBLIC_API_BASE_URL` 存在，则 `app/api-client.ts` 会按资源白名单代理到 Nest API。

## 数据库状态

当前存在两套 Prisma schema：

| 位置 | 数据库 | 用途 |
| --- | --- | --- |
| `prisma/schema.prisma` | SQLite | Web 本地 Next API routes |
| `apps/api/prisma/schema.prisma` | Postgres | Nest API |

当前开发阶段允许双轨存在，但生产方向应收敛到：

```text
Nest API + Postgres 为唯一业务后端
Web Next API routes 只保留兼容代理或删除
```

## 迁移优先级

建议按业务主线排序：

1. `knowledge-record` 单张 Agent 迁入 `/v1/agents/knowledge-record/run`。
2. 技术文章迁入 Nest API，作为知识来源材料的一类。
3. GitHub 趋势迁入 Nest API，便于做后台刷新和限流保护。
4. 面试文稿生成并入候选人准备或面试官 Agent。
5. 创业想法迁入 Nest API，作为非核心但可展示模块。
6. 导出能力迁入 Nest API，统一多用户和权限边界。

## 验收标准

- `NEXT_PUBLIC_API_BASE_URL=http://localhost:4000/v1` 时，核心链路走 Postgres 并可完整演示。
- 已迁移资源不再在 Next API routes 里继续扩写新业务逻辑。
- 所有 Agent 都能在 `AgentRunLog` 中看到输入、输出、模型、耗时、fallback 和错误。
- 文档中每个资源都有明确状态：已迁移、部分迁移、未迁移。
