## 当前判断

项目现在同时存在 Web 端、移动端和独立后端。

```text
view_agent
├── app/                 # 原 Next.js Web 端
├── components/          # 原 Web 组件
├── prisma/              # 原 Web/SQLite 数据模型
├── apps/
│   ├── api/             # NestJS 独立后端
│   └── mobile/          # Expo React Native 移动端
└── packages/
    └── shared/          # 共享 DTO、Zod schema、API client
```

Web 端目前仍然是原来的 Next.js 单体形态，适合继续快速验证功能。移动端已经升级到 Expo SDK 55，当前定位改成“可演示的纯前端 App”，不用后端也能直接打开。

同时，Web 端已经开始按资源代理到独立 Nest API：

```text
NEXT_PUBLIC_API_BASE_URL="http://localhost:4000/v1"
```

当这个环境变量存在时，`app/api-client.ts` 会把已迁移资源转发到 `/v1/*`；未迁移或未配置时仍走本地 Next API routes。因此当前是过渡架构，不是纯 Next，也不是纯 Nest。

## 产品路线

我们现在的主线不是单点功能堆叠，而是围绕“一个具体求职机会”持续推进准备：

1. 求职机会作为核心对象，串起公司、岗位、JD、简历、面试、复盘和下一步动作。
2. 知识库、技术文章、面经和开源趋势作为来源材料。
3. Agent 负责把来源材料生成准备内容、学习稿、匹配报告和模拟追问。
4. 面试官 Agent 和面试者练习负责训练。
5. 复盘卡和冲刺计划负责回写行动。
6. Web 端继续做主开发和产品调试。
7. 独立后端 `apps/api` 继续承接迁移，逐步成为唯一业务后端。
8. 移动端先保持精美演示版，等产品逻辑稳定后再接真实 API。

最终理想形态：

```text
Web 端：产品调试台 + 管理后台
Mobile：正式用户端
NestJS API：唯一业务后端
Postgres/Supabase：统一数据源
packages/shared：统一类型和接口契约
```

## 怎么启动移动端演示

演示移动端时，只需要启动 Expo：

```bash
cd /Users/william/coding/projects/view_agent
npx expo start -c
```

然后安卓手机打开 Expo Go，扫终端二维码。

注意：

- 手机里的 Expo Go 需要是 SDK 55 兼容版本。
- 现在移动端是纯前端演示，不需要先启动后端。
- 如果手机显示旧页面，先关掉 Expo Go 里的项目，再重新扫码。
- 如果终端提示端口占用，先停掉旧的 Expo 进程，再重新跑 `npx expo start -c`。

## 怎么启动 Web 端

如果只调 Web 本地 Next API + SQLite，可以启动：

```bash
cd /Users/william/coding/projects/view_agent
npm run dev:web
```

默认访问：

```text
http://localhost:3000
```

如果要拉起当前完整开发栈，建议直接用：

```bash
cd /Users/william/coding/projects/view_agent
npm run dev:all
```

它会依次处理：

- 启动 Docker Postgres
- 同步 Web SQLite schema
- 构建 shared types
- 同步 API Postgres schema
- 启动 Web 3000
- 启动 API 4000

注意：

- Web 根目录 `.env` 仍保留 `DATABASE_URL=file:./dev.db`，用于本地 Next API routes。
- `apps/api/.env` 使用 Postgres，默认是 `postgresql://interview:interview@localhost:54329/interview_ai?schema=public`。
- 如果 `.env` 配置了 `NEXT_PUBLIC_API_BASE_URL=http://localhost:4000/v1`，Web 会优先把已迁移资源代理到 Nest API。

## 现在怎么演示求职机会工作台

这是当前最重要的产品主线，建议优先演示：

1. 进入 Web 端 `求职机会`。
2. 新建一个机会，填写公司、岗位、级别、薪资、地点、来源和优先级。
3. 在详情页切到 `JD`，粘贴岗位描述并保存。
4. 点击刷新匹配，展示 included/missing keywords、匹配分和建议。
5. 切到 `简历版本`，创建定制版本。
6. 使用 `Auto-Select`，展示系统如何根据 JD 启用/重排内容块。
7. 对 missing keyword 生成候选 bullet，强调不会直接覆盖原始简历。
8. 切到 `活动`，展示创建、匹配、简历版本等动作都进入时间线。

这条链路表达的重点：

- 产品不只是 AI 问答，而是一个求职机会工作台。
- 简历、JD、匹配、版本、准备和行动都挂在同一个 Application 上。

## 现在怎么演示记录模块

记录模块现在已经升级为“题解式知识库”，建议按两条链路演示：

1. 进入 Web 端 `记录`。
2. 点击 `新增记录`。
3. 先演示 `单张整理`：粘贴一段零散八股，生成一篇结构化学习稿并保存。
4. 再切到 `文章拆卡`：
   - 可以从 `技术文章` 库选择一篇已保存文章。
   - 也可以直接粘贴一整篇技术文章或 Q&A 文本。
   - 设置最多生成张数，点击 `生成多篇学习稿`。
5. 展示多卡预览：
   - 每张卡会展示题解式预览。
   - 结构包含一句话结论、面试回答、考点定位、思路、步骤、示例、复杂度与取舍、易错点、常见追问、项目连接和复习清单。
   - 每张卡都能编辑题目、答案、主题、标签、题型、能力维度、难度和结构化学习稿 Markdown。
   - 可以取消勾选不想保存的卡。
   - 可以删除拆得不好的草稿。
6. 点击 `批量保存`，回到记录库确认多张八股卡已经进入现有题库。

这条链路的重点是：记录模块不再只是“贴一段存一张”，而是能把技术文章转换成可复习、可搜索、可面试复述、可追问的多篇题解式学习稿。

## 现在怎么演示 LeetCode 题库补齐

1. 进入 Web 端 `记录`。
2. 点击 `导入内置题库`。
3. 这次会一并导入两类内置题源：
   - 旧有的内部面试参考题。
   - 基于 LeetCode 官方公开 Interview / Study Plan 主题整理的原创改写卡。
4. 导入后可以在题库里直接看到算法基础、树图、动态规划、回溯、SQL 等常见面试主题。
5. 这些卡的写法保持我们的题解式结构，方便继续复习、追问和补充项目连接。

演示时要强调：

- 不是把 LeetCode 原文搬进来，而是把公开主题改写成我们自己的八股学习卡。
- 这样既补齐了算法基础题库，也不会破坏现有题库结构。

## 现在怎么演示面试官工作台

面试官模块现在建议按下面这条链路演示，而不是再把“面试官”讲成一份静态文稿：

1. 进入 Web 端 `面试官 Agent`。
2. 也可以先进入 Web 端 `模拟练习`，看顶部 `市场标杆补齐雷达`：
   - 它会对照主流平台能力，展示我们在个性化模拟、评分卡、题库、答案优化、coding 环境和真人压力模拟上的状态。
   - 点击每张卡的行动按钮，可以直接跳到对应模块。
3. 贴入简历，可选再贴 JD。
4. 选择时长和级别，启动 `面试官 Agent`。
5. 展示点：
   - 左侧一次出现完整主问题题纲。
   - 右侧是统一纪要区，不要求严格按顺序答题。
   - 任意主问题都可以回填纪要。
   - 回答不够完整时会自动挂出追问卡。
   - 偏离原题的讨论可以单独新建自由讨论卡。
   - 自由讨论卡还可以手动归属到某个主问题，归属后会并入该主问题复盘。
6. 结束后展示：
   - 主问题聚合复盘
   - 自由讨论复盘
   - 总分与 6 个维度分
   - 优势与下一步动作
7. 再回到“最近面试”里点开同一场面试，确认题纲和复盘都能恢复。
8. 展示“最近面试”卡片已经能区分 `面试官 Agent` 和旧模拟面试，并直接跳回对应工作区。

这条链路最能体现当前产品主线：

- 面试官 Agent 负责控场、追问、记录和评分。
- 面试者模块负责后续持续训练和复练。

这条链路表达的重点：

- 面试者不是单纯答题，而是先把简历变成可练习的表达材料。

## 现在怎么演示开源趋势

1. 进入 `开源趋势`。
2. 使用默认 `agent / AI Agent / 日榜 / 潜力分` 筛选。
3. 点击刷新，展示 GitHub 潜力榜。
4. 选中一个仓库，查看 star、fork、最近 push、潜力分和项目信号。
5. 点击 `AI 分析`，展示总结、趋势原因、学习价值、使用场景和风险。

7. 将仓库保存为来源材料，后续可供 Agent 引用。

- 无 `GITHUB_TOKEN` 时也能拉公开仓库，但更容易限流。
- GLM 失败时会返回规则 fallback，不应影响榜单展示。

## 之前遇到的问题和处理

安卓扫码没反应：

- 原因通常是手机访问不到电脑 IP，或者 Expo Go / SDK 版本不匹配。
- 我们把移动端升级到了 Expo SDK 55，和你的 Expo Go 对齐。

手机页面空白：

- 之前是移动端依赖后端数据，后端 `/v1/daily` 一度返回 500。
- 后端 500 的原因是开发用户 ID 和导入用户 email 不一致。
- 现在移动端已经改成纯前端演示版，不再因为后端没开而白屏。

Prisma generated client 找不到：

- 原因是 `nest build` 不会自动把 `src/generated/prisma` 复制到 `dist/src/generated/prisma`。
- 已经通过 Nest assets 和复制脚本修复。

## 下一步建议

短期：

1. 继续围绕 Application 打穿“准备 -> 模拟 -> 复盘 -> 下一步动作”闭环。
2. 把 `knowledge-record` Agent 接入统一 Agent 协议。
3. 给知识卡增加来源引用，让 Agent 输出可以追溯。

5. 优化首页，让“今天该做什么”直接来自当前高优先级 Application。

中期：

1. Web 端已迁移资源默认调用 `/v1/*`，Next API routes 只做兼容层。
2. 统一所有 Agent 的输入/输出/fallback/execution/evidence。
3. 技术文章、创业想法、开源趋势决定是否迁入 `apps/api`。
4. 移动端按模块接入真实 API，从题库、今日任务、模拟面试开始。

长期：

1. Supabase Auth + Postgres 成为统一登录和数据源。
2. Web 端成为管理后台和产品实验台。
3. 移动端成为正式用户端。
